Designing trust into a barcode-free checkout

Self Checkout+ is a computer vision grocery checkout system that identifies items without barcodes. I co-led the product direction and built the entire customer-facing experience, designing an interface that makes shoppers trust a system working invisibly behind the scenes.

01

The System

Traditional self-checkout fails at the same moment every time: when there's no barcode. Loose produce, bulk goods, specialty items, the customer either guesses a code from a printed menu or waves down an attendant. Self Checkout+ eliminates that failure point entirely.

A camera identifies grocery items visually through a YOLOv8 detection pipeline, matches them against a CLIP-trained embedding database, and adds them to a cart, no scanning required. The computer vision pipeline lives on a Raspberry Pi, connected to a custom ATmega328p MCU that handles hardware signaling.

The core challenge wasn't the computer vision. The CV pipeline was strong. The challenge was designing an interface that makes shoppers trust a system working entirely out of sight.

6 distinct customer screens in the checkout flow
2 user types: customer interface and admin dashboard
0 barcodes scanned in the entire checkout flow
4 person team across hardware, CV, and software

Key Design Decisions

The most interesting design work wasn't the visual styling, it was the product thinking that shaped the flow. Each decision below was a deliberate tradeoff I raised and resolved during the build.

01
Build in a confirmation step, don't auto-add items
When the CV pipeline identifies an item, a shopper sees "Did we get this right?" before it's added to the cart. This was a deliberate choice that trades a bit of speed for shopper trust. A system that silently adds items, even accurately, feels presumptuous. The explicit confirmation gives the customer a moment of control, and a clear path to correct errors without escalating to an attendant. The tradeoff is one extra tap per item; we decided that was worth it at this stage of trust-building with a novel checkout format.
UX decision
02
Cut the live video feed entirely
An early version showed a live camera feed in the scanning zone. I cut it. Live video is visually noisy, requires careful permission and privacy handling, and doesn't help the customer complete the transaction any faster. The task is checkout, not watching the camera process food. Removing the stream simplified the UI and shifted customer attention to where it belonged: the item confirmation and the cart. What replaced it was a clean scanning zone with a status indicator, enough to signal the system is active, nothing more.
Product decision
03
FastAPI as the single source of truth for cart totals
Early builds maintained a local React state for the cart and reconciled it with the backend periodically. This created subtle inconsistencies: the UI showed one running total while the backend calculated another. When they differed, even by a few cents, it looked like a bug, and it eroded trust in an interface where every number matters. I redesigned the data flow so the cart lives entirely in the backend and React simply renders what the API returns. One source of truth, no drift, no reconciliation bugs.
Engineering decision
04
Async handler pattern to prevent animation freezing
Framer Motion transitions would stutter and freeze when API calls blocked the main thread. The immediate fix was separating animation state from data fetching state: the UI plays the transition independently, and the component updates when the fetch resolves, not when it starts. This made every screen transition feel instant even when the backend was processing. It also made the architecture cleaner: view concerns and data concerns no longer competed on the same thread.
Engineering decision
05

The Integration Challenge

The hardest problems weren't in any single layer. They were at the seams between layers. The CV pipeline produced detection events. The FastAPI backend processed them into cart items. The React frontend rendered the cart. When all three systems were moving simultaneously, things broke in ways that no single component's tests could catch.

As the person responsible for the integration layer, I owned the SQLite schema that made the CV pipeline's output queryable by the frontend. I also designed the API contract between FastAPI and React, which fields the frontend could trust, which had to be calculated server-side, and how error states from the CV layer should surface to the customer.

The key insight

Integration isn't just a technical problem, it's a design problem. The decisions that prevented the most bugs weren't code changes; they were clarity about which layer owned which responsibility.

→
CV pipeline → SQLite → FastAPI → React
Each layer had a defined contract. The CV pipeline wrote detection events. SQLite stored item metadata. FastAPI computed totals. React rendered state. Keeping those responsibilities separate was what made the system debuggable.

The override queue was one of the most important features I designed, not because it was used often, but because it defined the graceful degradation path. When the CV pipeline failed, store staff needed a fast, auditable way to add items manually without disrupting the customer's session. Every override was logged with the reason code, giving us data on where the CV model needed improvement.

The Results

We wrote seven testable specs before building anything, and validated every one. The numbers I cared about most were the ones a shopper actually feels, that the total always matched, that the screen never froze, and that an uncertain item still resolved inside the same six-second window.

100% Top-1 accuracy on the 16-SKU test set, no misclassifications (target was 99 in 100)
<6s Full checkout, from item placement to emailed receipt, across 50 trials, none over 8s
12 hrs Continuous operation with zero crashes or disconnections
<200ms Admin operations, with no freezes during a live customer checkout

Two of those map straight onto the layer I owned. The cart total matched the backend on 50 of 50 transactions, because I'd made the backend the single source of truth instead of reconciling two running totals. And the interface stayed responsive mid-inference because animation and data-fetching ran on separate paths. Speed and accuracy are table stakes; the point was that trust held up under real conditions, including the moments the model wasn't sure.

08

What I Learned

01
Trust is a UX problem, not a technical one
The CV model was genuinely accurate. But accuracy doesn't equal trust. Trust comes from the interface: the confirmation step, the item preview, the clear error path, the ability to delete something. People trust systems that give them visible control, not systems that claim to be right.
02
Product decisions live in architecture meetings
Some of the most consequential product choices happened in technical discussions: whether to cache detection events or query live, how the backend should handle concurrent sessions, what the API contract between FastAPI and React should look like. Being in those conversations, and asking "what does this mean for the customer?", mattered more than any later feature decision.
03
Cut to clarify, not to simplify
Removing the live video feed wasn't about making the UI simpler. It was about making the customer's task clearer. Every element on screen should earn its place by helping the user complete their goal. The camera feed was technically impressive but functionally distracting, and impressive doesn't help someone check out faster.
04
Integration is a design problem
The hardest bugs weren't in any single component. They were at the seams. The animation and API call competing. The cart UI and backend total diverging. Designing clear ownership boundaries between layers would have caught those problems before they became debugging sessions. I'd now treat the API contract as a first-class design artifact, not an afterthought.

"The system could identify a banana with 99% confidence. My job was designing the 1%, the moment it doesn't, and what happens next."

Self Checkout+ taught me that the most interesting design work happens at the edge cases: when the technology fails gracefully, when the customer needs a clear way out, when the operator needs an audit trail. Getting the happy path right is table stakes. Getting the error path right is what defines the product.
